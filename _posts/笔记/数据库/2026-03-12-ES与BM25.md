## 简介

在看BM25时候，发现 Elasticsearch 默认就用 BM25 做相关性评分，而且查询速度很快，就算数据量很大也能毫秒级返回。这篇笔记记录一下 ES 里 BM25 是怎么用的，以及为什么查询不会慢。

ES（Elasticsearch）是基于 Lucene 构建的分布式搜索引擎，核心能力是全文检索。它从 5.0 版本开始把默认相似度算法从 TF-IDF 换成了 BM25，效果更好。

---

## 为什么查询不慢：倒排索引

ES 查询快的根本原因是**倒排索引（Inverted Index）**。

普通数据库的索引是正排：文档 -> 词。查"包含某个词的文档"，要扫所有文档，很慢。

倒排索引反过来：词 -> 文档列表。查某个词直接命中，O(1) 定位。

```
词          ->  文档列表（含位置信息）
"搜索"      ->  [doc1(pos:3), doc5(pos:1), doc9(pos:7)]
"引擎"      ->  [doc1(pos:4), doc3(pos:2)]
"BM25"      ->  [doc2(pos:1), doc5(pos:3), doc8(pos:5)]
```

具体存储的不只是文档 ID，还有：
- **Term Dictionary**：所有词的字典，按字母序排列，支持二分查找
- **Posting List**：每个词对应的文档 ID 列表（压缩存储）
- **词频（TF）**：词在每篇文档中出现的次数
- **位置信息**：词在文档中的位置（用于短语查询）

查询"BM25 搜索"时，只需要：
1. 在 Term Dictionary 里找"BM25"和"搜索"各自的 Posting List
2. 对两个列表求交集或合并
3. 用 BM25 对候选文档评分排序
4. 返回 TopN

整个过程基本不需要扫描原始文档，速度自然快。

## 文档插入过程

写入流程分几个阶段，理解这个有助于理解 ES 的实时性和持久性。

### 1. 写入内存缓冲区（Buffer）

文档先写入内存中的 **Index Buffer**，同时写 **Translog**（事务日志，类似 WAL）保证数据不丢失。

此时文档**不可搜索**。

### 2. Refresh：生成 Segment，数据可搜索

默认每隔 **1 秒** 执行一次 Refresh：
- 把 Index Buffer 里的文档构建成一个新的 **Segment**（内存中的倒排索引片段）
- Segment 写入文件系统缓存（OS Cache），**不是直接写磁盘**
- 此时文档**变为可搜索**

这就是为什么 ES 是"近实时"而不是"实时"——写入后约 1 秒才能搜到。

```
写入 -> Index Buffer -> [Refresh，每1s] -> Segment(OS Cache) -> 可搜索
```

### 3. Flush：Segment 持久化到磁盘

默认每隔 **30 分钟** 或 Translog 超过 512MB 时执行 Flush：
- 把 OS Cache 中的 Segment **fsync 刷入磁盘**
- 清空 Translog（数据已持久化，不再需要）

### 4. Segment Merge：合并小 Segment

随着写入积累，会产生大量小 Segment，影响查询性能（每次查询需要遍历所有 Segment）。

ES 后台会自动执行 **Merge**，把多个小 Segment 合并成一个大 Segment：
- 合并过程中会删除标记为删除的文档
- 合并完成后旧 Segment 删除，新 Segment 替换

```
Segment1 + Segment2 + Segment3  ->  MergedSegment
```

### 写入流程总结

```
客户端写入
    │
    ├─> Translog（持久化，防丢失）
    │
    └─> Index Buffer（内存）
            │
            └─[Refresh, 1s]─> Segment（OS Cache，可搜索）
                                    │
                                    └─[Flush, 30min]─> Segment（磁盘，持久化）
                                                            │
                                                            └─[Merge]─> 合并小Segment
```

## 搜索过程

搜索一个关键词的完整流程：

### 1. 查询分词

查询词先经过和建索引时相同的分词器处理。比如查"搜索引擎算法"，分词后得到 ["搜索", "引擎", "算法"]。

### 2. 倒排索引命中

对每个词，在 Term Dictionary 里查找对应的 Posting List，拿到候选文档集合。

多个词的结果根据查询类型合并：
- `must`（AND）：取交集
- `should`（OR）：取并集

### 3. BM25 评分

对每个候选文档，用 BM25 公式计算相关性得分：

$$\text{Score}(D, Q) = \sum_{i} \text{IDF}(q_i) \cdot \frac{f(q_i,D) \cdot (k_1+1)}{f(q_i,D) + k_1 \cdot (1-b+b \cdot \frac{|D|}{\text{avgdl}})}$$

ES 在建索引时就已经预计算并存储了每个词的 TF、文档长度等信息，评分时直接读取，不需要重新扫描文档内容。

### 4. 分布式聚合

在分布式场景下（多个 Shard），每个 Shard 独立计算本地的 TopN，然后 Coordinating Node 汇总所有 Shard 的结果，全局排序后返回最终 TopN。

```
Query -> Coordinating Node
            │
            ├─> Shard1: 本地BM25评分 -> 本地TopN
            ├─> Shard2: 本地BM25评分 -> 本地TopN
            └─> Shard3: 本地BM25评分 -> 本地TopN
                        │
                        └─> 汇总 -> 全局排序 -> 返回结果
```

## ES 中配置 BM25

### 查看默认相似度

ES 默认就是 BM25，默认参数 `k1=1.2`，`b=0.75`。

### 自定义 BM25 参数

可以在创建索引时自定义 k1 和 b：

```json
PUT /my_index
{
  "settings": {
    "similarity": {
      "my_bm25": {
        "type": "BM25",
        "k1": 1.5,
        "b": 0.8
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "similarity": "my_bm25"
      },
      "content": {
        "type": "text",
        "similarity": "my_bm25"
      }
    }
  }
}
```

### 全局修改默认相似度

```json
PUT /my_index
{
  "settings": {
    "index": {
      "similarity": {
        "default": {
          "type": "BM25",
          "k1": 1.5,
          "b": 0.75
        }
      }
    }
  }
}
```

### 搜索时查看评分解释

用 `explain` 可以看到每篇文档的 BM25 评分细节，调试很有用：

```json
GET /my_index/_search
{
  "explain": true,
  "query": {
    "match": {
      "content": "BM25 搜索引擎"
    }
  }
}
```

返回结果里会包含每个词的 IDF、TF、文档长度等细节，方便排查为什么某篇文档排名靠前或靠后。

### 混合检索（BM25 + 向量）

ES 8.x 支持 `knn` 向量检索，可以和 BM25 组合做混合检索：

```json
GET /my_index/_search
{
  "query": {
    "match": {
      "content": "搜索引擎"
    }
  },
  "knn": {
    "field": "content_vector",
    "query_vector": [0.1, 0.2, ...],
    "k": 10,
    "num_candidates": 100
  },
  "rank": {
    "rrf": {}
  }
}
```

`rrf`（Reciprocal Rank Fusion）会自动把 BM25 结果和向量检索结果融合排序，不需要手动调权重。

## 总结

ES 查询快的核心是倒排索引，查找词直接命中，不需要全表扫描。BM25 在评分阶段也用到了预存的 TF、文档长度等信息，不需要实时计算，所以整体很高效。

写入流程的几个关键节点：
- **Refresh（1s）**：数据变为可搜索，写入 OS Cache
- **Flush（30min）**：数据持久化到磁盘，清空 Translog
- **Merge**：后台合并小 Segment，保持查询性能

实际用 ES 做 RAG 混合检索体验不错，BM25 负责关键词精确匹配，向量检索负责语义理解，两者用 RRF 融合，召回效果比单独用任何一种都好。
